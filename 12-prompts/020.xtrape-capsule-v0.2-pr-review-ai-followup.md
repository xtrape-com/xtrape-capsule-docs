### Review Status


### Positive Findings

CE #16 now satisfies the previous review requirements.

### Effective Status Semantics

`deriveEffectiveStatus` now correctly follows the recommended semantics:

```ts
if (!agent) return "OFFLINE";
if (agent.status === "REVOKED" || agent.status === "DISABLED") return "OFFLINE";
if (agent.status === "OFFLINE") return "OFFLINE";

const lastHbMs = agent.lastHeartbeatAt ? Date.parse(agent.lastHeartbeatAt) : 0;
if (!lastHbMs || nowMs - lastHbMs > offlineThresholdSeconds * 1000) return "STALE";

return row.status;
```

This is correct:

```text
STALE = agent row is not yet OFFLINE, but heartbeat freshness has expired.
OFFLINE = agent missing / disabled / revoked / already OFFLINE.
```

Service responses also expose:

```text
status       = effective status
storedStatus = DB stored service status
```

This is the right product/API shape.

### Version Fallback

CE now uses:

```ts
const BACKEND_FALLBACK_VERSION = "0.2.0-dev";
```

This fixes the previous problem where local dev could still report `0.1.0`.

### Command Failure Surface

CE now surfaces command failure reason on the command row:

```text
errorCode
errorMessage
durationMs
```

It extracts failure fields from the agent-reported result:

```text
body.error.code
body.error.message
body.message
```

This is aligned with Demo #14's structured `failOnce` behavior.

### DB Schema Check

The `commands` table already includes:

```sql
errorCode text,
errorMessage text
```

Therefore the CE command-row surfacing is not missing schema support.

### Acceptance Criteria

- [x] Effective status semantics are correct.
- [x] `0.2.0-dev` fallback is in place.
- [x] `/api/system/health` and `/api/system/version` shapes are implemented.
- [x] `/health` is implemented.
- [x] Command failure surfacing is implemented.
- [x] `durationMs` is implemented.
- [x] Admin metrics shape is enriched.
- [x] No release-blocking concern remains.
