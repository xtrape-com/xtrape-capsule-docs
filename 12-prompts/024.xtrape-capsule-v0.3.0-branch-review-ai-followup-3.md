## 2. Authoritative v0.3 Baseline

Use this terminology consistently:

```text
agent-node
  = Node.js Embedded Agent SDK.
  = Keep this repository/package name.

node agent / Node Agent
  = old name for node-level runtime.
  = Replace with OpHub.

OpHub
  = node-level local operations hub runtime.
  = Connects to Opstage and manages one or more local Capsule Services.

OpHub-managed mode
  = Capsule Service does not connect directly to Opstage.
  = Capsule Service exposes local adapter endpoints.
  = OpHub discovers, reports, routes, and forwards results for it.

OpHub Self Service
  = OpHub reports itself as an infrastructure/system Capsule Service.
  = Implemented through a built-in adapter inside OpHub.
```

Expected version theme:

```text
v0.3 OpHub Runtime, Capsule Events and Capability Metadata
```

Expected architecture:

```text
Embedded Agent mode:
  Opstage CE <----polling---- Capsule Service
                                embeds agent-*

OpHub-managed mode:
  Opstage CE <----polling---- OpHub <----local adapter---- Capsule Service A
                                   <----local adapter---- Capsule Service B

OpHub Self Service:
  OpHub reports itself as an infrastructure/system service through a built-in adapter.
```

## MAJOR-1: Demo adapter mode now reuses actions, but service code is hardcoded as `demo-adapter`

### Repository

```text
xtrape-capsule-demo
```

### Branch

```text
v0.3.0
```

### File

```text
src/server.ts
```

### Evidence

`server.ts` now creates adapter actions with:

```ts
const DEMO_ACTIONS = createDemoActions("demo-adapter");
```

This is much better than the previous hardcoded `demo.echo` / `demo.job` / `demo.status` action set because it now reuses the same action definitions and handlers from embedded mode.

### Concern

The manifest is created from:

```ts
const config = demoServiceConfigFromEnv();
const manifest = createManifest(config, "adapter");
```

So the service code may be:

```text
DEMO_SERVICE_CODE
```

defaulting to `demo-worker`.

But the action handlers use service code:

```text
demo-adapter
```

This can create inconsistent action outputs and prepared state, especially for `echo`, where the response includes `service: serviceCode`.

### Required fix

Create actions using the same service code as the manifest:

```ts
function createAdapterActions() {
  const config = demoServiceConfigFromEnv();
  return createDemoActions(config.code);
}
```

or initialize `DEMO_ACTIONS` after reading config.

Avoid module-level hardcoded service code.

### Acceptance criteria

- [ ] Adapter action handlers use the same service code as `/capsule/manifest`.
- [ ] `GET /capsule/actions` returns same action names as embedded mode.
- [ ] `POST /capsule/actions/echo/prepare` returns currentState.service matching the configured service code.
- [ ] `POST /capsule/actions/echo/execute` returns service matching the configured service code.
- [ ] `pnpm typecheck` and `pnpm build` pass.

## MAJOR-2: CE still uses local registration schema instead of contract schema

### Repository

```text
xtrape-capsule-ce
```

### Branch

```text
v0.3.0
```

### File

```text
apps/opstage-backend/src/app.ts
```

### Evidence

CE now correctly uses:

```ts
mode: z.enum(["embedded", "ophub"]).default("embedded")
```

This aligns with contracts.

However, CE still defines:

```ts
const v03RegisterAgentRequestSchema = z.object(...)
```

instead of importing and using:

```ts
registerAgentRequestSchema
```

from `@xtrape/capsule-contracts-node`.

### Current assessment

Not a blocker because the enum now matches.  
Still a drift risk.

### Recommended fix

Prefer to use the contracts schema directly:

```ts
import { registerAgentRequestSchema } from "@xtrape/capsule-contracts-node";
```

Then in the route:

```ts
const body = registerAgentRequestSchema.parse(req.body);
```

If CE intentionally keeps a local schema, add a comment explaining why it differs from the public contract.

### Acceptance criteria

- [ ] CE registration schema is either imported from contracts or explicitly documented as a CE-local extension.
- [ ] CE tests cover `mode: "ophub"`.
- [ ] CE tests reject unsupported mode values such as `node` and `sidecar`.

## MAJOR-4: Contracts still lack explicit OpHub Self Service classification

### Repository

```text
xtrape-capsule-contracts-node
```

### Branch

```text
v0.3.0
```

### Observation

Contracts now correctly define:

```ts
AgentMode = ["embedded", "ophub"]
```

and support capabilities/events/multi-service reports.

However, `ReportedServiceSchema` does not explicitly define how to represent:

```text
OpHub Self Service
business child service
ordinary embedded service
```

### Current assessment

Not a blocker for MVP if OpHub encodes this in `manifest`, but the convention should be documented.

### Recommended fix

Either add formal fields:

```ts
export const ServiceKind = ["business", "infrastructure", "system"] as const;
```

and include it in `ReportedServiceSchema`, or document a manifest convention:

```json
{
  "xtrape": {
    "serviceKind": "infrastructure",
    "managedBy": "ophub"
  }
}
```

### Acceptance criteria

- [ ] OpHub Self Service can be represented explicitly or through a documented convention.
- [ ] OpHub README/demo examples use the same convention.
- [ ] CE can display or at least preserve the distinction.
