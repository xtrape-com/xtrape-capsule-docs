# Xtrape Capsule Public Review Consistency Fix Prompt

> Purpose: Give this prompt to an AI coding agent such as Codex or Claude Code to apply a focused cross-repository cleanup for the current Xtrape Capsule Public Review state.
>
> Scope: Documentation and small consistency fixes only. Do **not** introduce new features, new runtime behavior, new marketplace implementation, or broad refactors.

---

## 1. Background

Xtrape Capsule is currently preparing for Public Review / pre-`v0.1.0 Public Preview`.

The current repository set is:

```text
xtrape-capsule-ce
xtrape-capsule-demo
xtrape-capsule-site
xtrape-capsule-agent-node
xtrape-capsule-contracts-node
xtrape-capsule-docs
```

Recent work has significantly improved Public Review readiness:

- CE has command lifecycle race fixes, safer agent-token rotation, better session/cookie defaults, sampled heartbeat audit, and improved audit redaction.
- Agent SDK has improved redaction, `agent-token.txt` token-store direction, README troubleshooting, and explicit optional `agent` metadata in examples.
- Contracts now remove the old bare `ACTION` command type and keep only `ACTION_PREPARE` / `ACTION_EXECUTE`.
- Site has a Public Review announcement and v0.1.0 release draft.
- Demo now has a richer Capsule Service sample with list/detail/row-action examples.

However, several public-facing docs still contain inconsistent wording or old protocol references.

This task is to clean those up before Public Review.

## 3. Required Fixes

## 3.1 `xtrape-capsule-demo`: remove old `ACTION` compatibility wording

### Problem

`xtrape-capsule-contracts-node` now defines command type as:

```text
ACTION_PREPARE
ACTION_EXECUTE
```

The old bare `ACTION` command type has been removed from the public contract.

But `xtrape-capsule-demo/README.md` still contains a table row similar to:

```text
ACTION | Compatibility command shape handled by the SDK as an execute request.
```

### Required change

In `xtrape-capsule-demo/README.md`:

1. Remove the old `ACTION` compatibility row.
2. Make the action flow section explicitly state that the current public protocol uses only:

```text
ACTION_PREPARE
ACTION_EXECUTE
```

Suggested wording:

```md
The current public protocol uses two command types for Capsule actions:

- `ACTION_PREPARE` — Opstage asks the service for initial input, current state, and optional dynamic action metadata before execution.
- `ACTION_EXECUTE` — Opstage asks the service to run an action with a submitted payload.

Older `ACTION`-style wording has been removed from the public contract and should not appear in Public Review documentation.
```

### Acceptance

- No `ACTION` compatibility row remains in the demo README.
- The README does not imply that bare `ACTION` is a valid current command type.

```text
./data/agent-token.txt
```

```text
/app/data/agent-token.txt
```

Check and update at least:

```text
.env.example
README.md
docker-compose.yml
src/** if applicable
```

### Acceptance

- Demo README, `.env.example`, Docker Compose, and source defaults agree on `.txt`.
- No public demo instruction recommends `agent-token.json` as the modern path.
- If legacy `.json` support exists in the SDK, it may remain mentioned only as legacy compatibility in SDK docs/tests, not as the demo default.

## 3.4 `xtrape-capsule-site`: unify Docker image wording

### Problem

Current public docs use mixed wording:

- CE README may say GHCR images are available.
- Release note may say `0.1.0` image is planned.
- Public Review announcement may say Docker images are built on every push to `main`.

This confuses users about what is canonical today.

### Required decision

Use this layered wording unless actual project maintainers choose otherwise:

```text
Public Review:
  Source build via Docker Compose is the canonical path.
  GHCR pre-release/main images may exist, but should not be the only documented path.

Public Preview v0.1.0:
  ghcr.io/xtrape-com/xtrape-capsule-ce:0.1.0 becomes the canonical image path after the v0.1.0 cut.

latest:
  Do not make latest the main recommended path during Public Review.
```

### Required changes

Check and update:

```text
xtrape-capsule-ce/README.md
xtrape-capsule-site/docs/releases/v0.1.0.md
xtrape-capsule-site/docs/public-review-announcement.md
xtrape-capsule-site/docs/getting-started/quick-start.md if relevant
xtrape-capsule-site/docs/getting-started/install-opstage-ce.md if relevant
```

### Suggested wording for Public Review

```md
During Public Review, the source-build Docker Compose path remains the canonical quick-start path. GHCR images may be produced from `main` for validation, but the stable `ghcr.io/xtrape-com/xtrape-capsule-ce:0.1.0` image becomes the primary documented path only after the `v0.1.0 Public Preview` cut.
```

### Acceptance

- Docs no longer contradict themselves about whether `0.1.0` image is already stable.
- Public Review path and Public Preview path are clearly separated.
- `latest` is not the main recommended command in Public Review docs.

### Problem

```ts
import { ActionPrepareResultSchema } from "@xtrape/capsule-contracts-node";

const prepare = ActionPrepareResultSchema.parse({
  initialPayload: { message: "hello" },
  currentState: { service: "ready" },
  inputSchema: {
    type: "object",
    properties: {
      message: { type: "string", default: "hello" },
    },
  },
});
```

Explain that when reported back to Opstage, this shape is placed under `ReportCommandResultRequest.data` for `ACTION_PREPARE`.

### Acceptance

- README directly names and uses `ActionPrepareResultSchema`.
- It remains clear that command result reporting uses `ReportCommandResultRequestSchema`.
