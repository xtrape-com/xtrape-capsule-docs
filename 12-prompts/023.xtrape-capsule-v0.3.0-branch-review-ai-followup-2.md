## 2. Authoritative v0.3 Baseline

Use this terminology consistently:

```text
agent-node
  = Node.js Embedded Agent SDK.
  = Keep this repository/package name.

node agent / Node Agent
  = old name for node-level runtime.
  = Replace with OpHub.

OpHub
  = node-level local operations hub runtime.
  = Connects to Opstage and manages one or more local Capsule Services.

OpHub-managed mode
  = Capsule Service does not connect directly to Opstage.
  = Capsule Service exposes local adapter endpoints.
  = OpHub discovers, reports, routes, and forwards results for it.

OpHub Self Service
  = OpHub reports itself as an infrastructure/system Capsule Service.
  = Implemented through a built-in adapter inside OpHub.
```

Expected architecture:

```text
Embedded Agent mode:
  Opstage CE <----polling---- Capsule Service
                                embeds agent-*

OpHub-managed mode:
  Opstage CE <----polling---- OpHub <----local adapter---- Capsule Service A
                                   <----local adapter---- Capsule Service B

OpHub Self Service:
  OpHub reports itself as an infrastructure/system service through a built-in adapter.
```

Expected version theme:

```text
v0.3 OpHub Runtime, Capsule Events and Capability Metadata
```

## 3. Executive Review Conclusion

The latest `v0.3.0` branches have improved significantly, but they are **still not ready**.

Current status:

```text
v0.3.0 readiness: Not ready
Architecture terminology: mostly corrected
Contracts/CE AgentMode: fixed
Demo adapter endpoints: partially implemented
OpHub runtime implementation: not verifiable
Public roadmap: inconsistent
```

Improvements since the previous review:

- CE and contracts now both use `embedded | ophub` as AgentMode.
- CE fallback version is now `0.3.0-dev`.
- Demo has added `/capsule/*` adapter endpoints.
- Internal docs roadmap now says `OpHub Runtime` instead of `Node Agent`.
- Site current status now says `Node.js Embedded Agent SDK`.
- OpHub README says it registers using `mode: "ophub"` and documents OpHub Self Service.

Remaining blockers:

1. `ophub-go` still cannot be verified as an actual buildable Go runtime; README says `go build ./cmd/ophub`, but `cmd/ophub/main.go` was not found.
2. Demo adapter endpoints exist, but they do not reuse the existing `createDemoActions()` action definitions/handlers and use different action names (`demo.echo`, `demo.job`, `demo.status`) from Embedded Agent mode (`echo`, `runDemoJob`, `setHealthMode`, etc.).
3. `agent-node` still contains an empty `src/node-agent.ts` file.
4. Site roadmap snapshot is now internally inconsistent: it introduces `v0.4 Production Preview` and `v1.0 General Availability` while also retaining the older v0.5-v0.7 and `v1.0 CE Stable` sequence.
5. Demo still adds `docs/v0.3-node-agent-events-and-capabilities.md`, which conflicts with the new OpHub terminology.

## BLOCKER-1: `ophub-go` build/runtime implementation is still not verifiable

### Repository

```text
xtrape-capsule-ophub-go
```

### Branch

```text
v0.3.0
```

### Evidence

README says:

```bash
go build ./cmd/ophub
```

and describes OpHub as a runtime that registers using `mode: "ophub"`, reports OpHub Self Service, discovers local services, polls commands, and routes commands.

However, during review:

- `cmd/ophub/main.go` could not be fetched.
- repository search did not find `package main`, `func main`, or `cmd/ophub`.
- only `go.mod` and README-level evidence were confirmed.

### Why this blocks v0.3

OpHub is the central v0.3 runtime. If the documented binary does not exist, the v0.3 implementation cannot be accepted.

### Required fix

Ensure actual implementation exists and is buildable.

Minimum expected structure:

```text
cmd/ophub/main.go
internal/config/
internal/opstage/
internal/runtime/
internal/adapters/http/
internal/services/
internal/commands/
internal/security/
```

Minimum runtime behavior:

- load `ophub.yaml`;
- register with Opstage using `mode: "ophub"`;
- persist OpHub token;
- send heartbeat;
- report OpHub Self Service;
- discover/report child services through local HTTP adapter;
- poll commands;
- route commands to the correct local service;
- report command results.

### Acceptance criteria

- [ ] `cmd/ophub/main.go` exists.
- [ ] `go build ./cmd/ophub` passes.
- [ ] `go test ./...` passes.
- [ ] OpHub registers with CE using `mode: "ophub"`.
- [ ] OpHub reports `ophub-self` or configured self service.
- [ ] OpHub reports at least one child service.
- [ ] OpHub can call demo `/capsule/*` endpoints.
- [ ] README build/run instructions match actual files.

## BLOCKER-2: Demo adapter endpoints exist but are not aligned with embedded demo actions

### Repository

```text
xtrape-capsule-demo
```

### Branch

```text
v0.3.0
```

### Files

```text
src/server.ts
src/actions.ts
README.md
```

### Evidence

`src/server.ts` now implements:

```text
GET  /capsule/manifest
GET  /capsule/health
GET  /capsule/configs
GET  /capsule/actions
POST /capsule/actions/:name/prepare
POST /capsule/actions/:name/execute
```

This resolves the previous “missing endpoint” blocker.

However, adapter mode defines a separate `DEMO_ACTION_DEFINITIONS` list using:

```text
demo.echo
demo.job
demo.status
```

Meanwhile, embedded mode still uses `createDemoActions()` with names such as:

```text
echo
runDemoJob
setHealthMode
searchDemoItems
createDemoItem
inspectDemoItem
runDemoJobForItem
disableDemoItem
failOnce
```

### Problem

Adapter mode and Embedded Agent mode now demonstrate different action surfaces.

This weakens the demo as the canonical v0.3 reference because:

- OpHub-managed mode does not validate the same action set as Embedded Agent mode.
- Existing smoke tests and docs may not map cleanly.
- Opstage commands created for action names from one mode will not match the other mode.
- OpHub E2E cannot validate the richer action patterns already present in `createDemoActions()`.

### Required fix

Make adapter mode reuse the existing demo actions as much as practical.

Recommended approach:

1. Refactor `createDemoActions(serviceCode)` so its definitions and handlers can be called by both:
   - Embedded Agent SDK registration;
   - local HTTP adapter endpoints.

2. `/capsule/actions` should return the same action definitions as Embedded Agent mode.

3. `/capsule/actions/:name/prepare` should call the matching action's `prepare` function if present.

4. `/capsule/actions/:name/execute` should call the matching action's `handler`.

5. Keep behavior deterministic enough for smoke tests.

### Acceptance criteria

- [ ] Adapter mode reports the same action names as embedded mode, unless intentionally documented otherwise.
- [ ] Adapter mode supports at least `echo`, `runDemoJob`, `setHealthMode`, and `failOnce`.
- [ ] Adapter mode can validate structured failure through `failOnce`.
- [ ] Adapter mode can validate command duration through `runDemoJob`.
- [ ] README and smoke docs use action names that actually exist in adapter mode.
- [ ] `pnpm typecheck` and `pnpm build` pass.

## MAJOR-1: Contracts still do not explicitly model OpHub Self Service kind

### Repository

```text
xtrape-capsule-contracts-node
```

### Observation

Contracts now correctly define:

```ts
AgentMode = ["embedded", "ophub"]
```

and define capability/event metadata plus service report schemas.

However, `ReportedServiceSchema` still does not have an explicit `serviceKind` / `managedBy` field.

### Concern

Without a standard field, CE and UI cannot reliably distinguish:

```text
OpHub Self Service
OpHub child business service
ordinary Embedded Agent service
```

### Recommended fix

Either add a formal enum:

```ts
export const ServiceKind = ["business", "infrastructure", "system"] as const;
```

and include it in `ReportedServiceSchema`, or document a metadata convention:

```json
{
  "xtrape": {
    "serviceKind": "infrastructure",
    "managedBy": "ophub"
  }
}
```

### Acceptance criteria

- [ ] OpHub Self Service can be represented explicitly or through a documented convention.
- [ ] OpHub README/demo examples use the same convention.
- [ ] CE can display the distinction if needed.
