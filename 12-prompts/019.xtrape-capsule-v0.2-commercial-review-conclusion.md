# Xtrape Capsule v0.2 Commercial Product Review Conclusion

> Review target: `xtrape-capsule` v0.2 release train  
> Review scope: CE PR #16, agent-node PR #7, contracts-node PR #9, site PR #6  
> Review standard: commercial product readiness, public preview quality, cross-repository release consistency  
> Recommended next executor: Codex / Claude Code / other AI coding agent  
> Review date: 2026-05-14

---

## 1. Executive Conclusion

The v0.2 release direction is valid and commercially meaningful.

The release theme, **Developer Experience & Runtime Maturity**, is appropriate for moving Xtrape Capsule from `v0.1 Public Review` toward `v0.2 Public Preview`.

However, under a commercial product standard, the current release train should **not be merged and published as-is**. It should first be treated as:

> `v0.2.0-rc.1`

The main reason is not that the implementation direction is wrong, but that the release train has several **cross-repository consistency gaps**:

1. The CE runtime implementation and site documentation are not fully aligned.
2. The site release notes appear to claim some capabilities that are not clearly implemented in the listed PR set.
3. The version matrix and release order are not yet fully locked.
4. Some manual smoke checks are still marked incomplete.
5. The semantics of `effectiveStatus` require final product-level confirmation.

Recommended decision:

> **Do not directly merge all PRs yet. Finish the release consistency fixes, then publish v0.2 as Public Preview.**


### Main Value

This PR is the core of v0.2. It adds or improves:

- Runtime `effectiveStatus`.
- `/api/system/health`.
- `/api/system/version`.
- `/health`.
- Docker image OCI metadata.
- `/api/admin/metrics` with command duration and top error data.
- Command row `durationMs`, `errorCode`, `errorMessage`.
- Reduced heartbeat write noise.
- UI refactor from large `App.tsx` into modular pages/components.
- ADR for ephemeral action secrets.
- Removal of shared `newId()` export.

### Commercial Assessment

This PR provides real operator value and should be considered close to mergeable.

The strongest commercial improvements are:

1. Operators can see more accurate service status.
2. Operators can see why commands failed without deep inspection.
3. Health/version endpoints improve deployability.
4. Metrics make the product more credible for private deployment.
5. UI modularization improves maintainability for future commercial editions.

### Blocking Concern: `effectiveStatus` Semantics

Current logic appears to map `agent.status === "OFFLINE"` to service `STALE`.

This needs product-level confirmation.

There are two possible interpretations:

### Option A: Recommended

Use:

```text
STALE = heartbeat freshness has expired, but maintenance has not yet finalized offline state.
OFFLINE = agent is already known offline / disabled / revoked / missing.
```

Then CE should return `OFFLINE` when agent row is already `OFFLINE`.

Recommended logic:

```ts
if (!agent) return "OFFLINE";
if (agent.status === "REVOKED" || agent.status === "DISABLED") return "OFFLINE";
if (agent.status === "OFFLINE") return "OFFLINE";

const lastHbMs = agent.lastHeartbeatAt ? Date.parse(agent.lastHeartbeatAt) : 0;
if (!lastHbMs || nowMs - lastHbMs > offlineThresholdSeconds * 1000) return "STALE";

return row.status;
```

### Option B: Accept Existing Behavior

If the product wants `STALE` to mean “agent/service not currently fresh or reachable”, then current behavior may be acceptable. But the documentation must explicitly say that `OFFLINE` is reserved for stronger non-operational states such as revoked, disabled, or missing agent.

### Recommendation

Choose Option A. It is easier for operators to understand.

### Blocking Concern: Docker Tag Policy

The Docker workflow now appears to prefer:

- `edge` for main branch.
- branch ref tag.
- semver tag.
- `sha-*`.

This is reasonable, but the product must clearly define whether `latest` is published.

Commercial recommendation:

- Use `0.2.0` as the primary documented tag.
- Use `edge` for main/nightly builds.
- Use `sha-*` for traceable deployments.
- Avoid documenting `latest` unless it is intentionally produced and supported.

### Required Before Merge

- [ ] Confirm and fix/document `effectiveStatus` semantics.
- [ ] Confirm Docker tag policy.
- [ ] Ensure `/api/system/version` fallback is not misleading. Prefer `0.2.0-dev` or build-time injected version instead of `0.1.0`.
- [ ] Complete manual smoke tests:
  - [ ] Services drawer opens correctly.
  - [ ] Action prepare detail renders correctly.
  - [ ] Action execution result list renders correctly.
  - [ ] `/health` returns expected shape.
  - [ ] `/api/system/health` returns expected shape.
  - [ ] `/api/system/version` returns expected shape.
  - [ ] `/api/admin/metrics` returns expected shape.

### Suggested Merge Decision


## 3.3 `xtrape-capsule-contracts-node` PR #9

PR:

```text
https://github.com/xtrape-com/xtrape-capsule-contracts-node/pull/9
```

Title:

```text
v0.2 remove dead newId() helper
```

### Main Value

This PR removes:

- `newId()`
- `idPrefixes`
- `IdPrefix`
- `nanoid` runtime dependency

### Commercial Assessment

This change is architecturally correct.

A contracts package should provide wire schemas, types, constants, and validation rules. It should not provide runtime ID generation unless that is a formal part of the wire contract.

Removing `newId()` reduces confusion and avoids mismatch between client-side ID generation and backend ID generation.

### Breaking Change

This is an intentional breaking change for any external code importing `newId`, `idPrefixes`, or `IdPrefix`.

Because the package is still `0.x`, this is acceptable, but it must be clearly documented.

### Required Before Merge

- [ ] Ensure package version is bumped to `0.2.0`.
- [ ] Ensure release notes clearly state the breaking change.
- [ ] Ensure CE and agent-node do not import removed symbols.
- [ ] Ensure README no longer references removed helpers.
- [ ] Ensure consumers are told to provide their own ID factory if needed.

### Suggested Merge Decision

```text
Merge after version and release-note confirmation.
```
