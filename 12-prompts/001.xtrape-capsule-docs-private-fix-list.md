# xtrape-capsule-docs Private Design Repository Fix List

## Purpose

`xtrape-capsule-docs` will be converted back to a **private** repository.

This repository is **not** the public product website and should not be optimized as a marketing/public documentation site.

Its intended role is:

> Private design documentation, architecture decisions, specification drafts, implementation guidance, and AI coding agent context for the Xtrape Capsule product family.

The public-facing site is:

```text
xtrape-capsule-site
```

The private design repository should act as:

```text
Source of Design Truth
```

while the public site should act as:

```text
Public Product Narrative
```

---

## Review Scope

Repository:

```text
xtrape-capsule-docs
```

Primary goals:

```text
1. Make it maintainable as a long-term private design knowledge base.
2. Make it easy for AI coding agents to read and follow.
3. Separate design drafts, accepted decisions, implementation guidance, and future roadmap.
4. Avoid confusing it with public product documentation.
5. Enable controlled extraction of stable content into xtrape-capsule-site.
```

## P0-3. Reformat Markdown Files

### Problem

Several Markdown files appear compressed into very long single-line content. This makes the repository hard to review, hard to diff, and hard for AI coding agents to edit safely.

### Required Action

Reformat all Markdown files into normal readable Markdown.

Prioritize:

```text
README.md
README_zh.md
*/README.md
*/README_zh.md
01-capsule/**/*.md
02-specs/**/*.md
03-editions/**/*.md
04-opstage/**/*.md
05-agents/**/*.md
06-runtimes/**/*.md
07-roadmap/**/*.md
08-decisions/**/*.md
09-contracts/**/*.md
10-implementation/**/*.md
```

Formatting rules:

```text
1. One heading per line.
2. Blank line before and after headings.
3. Blank line before and after fenced code blocks.
4. Markdown lists should use one item per line.
5. Tables should be readable in raw GitHub view.
6. Avoid very long lines where possible.
7. Do not compress entire sections into a single line.
```

### Acceptance Criteria

```text
[ ] README.md is readable in raw GitHub view.
[ ] Major docs are readable in raw GitHub view.
[ ] Git diffs become meaningful.
[ ] AI/code agent can edit individual sections safely.
```

# P1 - Strongly Recommended

## P1-1. Standardize ADR Format

### Problem

ADR documents are valuable but should use a consistent structure.

### Required Action

For every ADR in:

```text
08-decisions/
```

Use this template:

```md
# ADR-000X: Title

## Status

Accepted / Proposed / Superseded / Deprecated

## Date

YYYY-MM-DD

## Context

What problem or decision pressure led to this decision?

## Decision

What was decided?

## Consequences

What becomes easier?
What becomes harder?
What trade-offs are accepted?

## Alternatives Considered

What options were considered and rejected?

## Implementation Notes

Where is this decision reflected in code/docs?

## Supersedes / Superseded By

Optional.
```

### Acceptance Criteria

```text
[ ] All ADRs use the same structure.
[ ] Each ADR has a clear status.
[ ] Accepted ADRs are distinguishable from proposed ADRs.
[ ] If an ADR is superseded, the replacement is linked.
```

## P1-5. Add Public Extraction Guide

### Problem

Some private design content may later be rewritten into public website content. There should be a safe extraction process.

### Required Action

Create:

```text
PUBLIC_EXTRACTION_GUIDE.md
```

Suggested content:

```md
# Public Extraction Guide

## Purpose

This guide defines how stable private design content can be rewritten into public product documentation.

## Public Target

xtrape-capsule-site

## Rules

1. Do not copy internal planning text directly.
2. Remove commercial planning details unless approved.
3. Remove patent, moat, pricing, security-sensitive, and internal roadmap details.
4. Rewrite design language into user-facing language.
5. Prefer examples using:
   - integration services
   - automation workers
   - background jobs
   - agent runtimes
6. Do not mention CAPI in public docs unless explicitly approved.
7. Public docs should be concise, actionable, and stable.

## Extraction Checklist

[ ] Source doc status is accepted or implemented.
[ ] No internal-only business content remains.
[ ] No sensitive implementation details remain.
[ ] Public wording is aligned with xtrape-capsule-site positioning.
[ ] Links are updated to public URLs.
```

### Acceptance Criteria

```text
[ ] PUBLIC_EXTRACTION_GUIDE.md exists.
[ ] It explains how to move content to xtrape-capsule-site.
[ ] It includes safety rules for internal/commercial/sensitive content.
```
