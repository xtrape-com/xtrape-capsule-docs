## 2. Expected v0.3 Architecture Baseline

The expected v0.3 theme is:

```text
OpHub Runtime, Capsule Events and Capability Metadata
```

Terminology baseline:

```text
agent-node
  = Node.js Embedded Agent SDK repository/package.
  = Keep this name.

node agent / Node Agent
  = old name for node-level runtime.
  = Must be replaced by OpHub.

OpHub
  = node-level local operations hub runtime.
  = Can run as service / sidecar / Docker runtime.
  = Manages one or more local Capsule Services.

OpHub Self Service
  = OpHub itself reported as an infrastructure/system Capsule Service.
  = Should be implemented through a built-in adapter inside OpHub.
  = Should not be implemented by embedding agent-go or agent-node.
```

Correct architecture:

```text
Embedded Agent mode:
  Opstage CE <----polling---- Capsule Service
                                embeds agent-*

OpHub-managed mode:
  Opstage CE <----polling---- OpHub <----local adapter---- Capsule Service A
                                   <----local adapter---- Capsule Service B

OpHub Self Service:
  OpHub reports itself as an infrastructure/system service through a built-in adapter.
```

Expected dependency/boundary model:

```text
contracts-*:
  language-specific representation of the Capsule wire protocol

agent-*:
  language-specific Embedded Agent SDKs for business Capsule Services

ophub-go:
  standalone multi-service local runtime
```

`ophub-go` should not depend conceptually on `agent-go`. If shared Go client code is needed later, extract a lower-level client package rather than making OpHub depend on an embedded SDK.

## BLOCKER-1: CE v0.3 register schema accepts `node`, but contracts use `ophub`

### Repository

```text
xtrape-capsule-ce
```

### File

```text
apps/opstage-backend/src/app.ts
```

### Evidence

CE defines a local `v03RegisterAgentRequestSchema` with:

```ts
mode: z.enum(["embedded", "node", "sidecar", "external"]).default("embedded")
```

This local schema is inconsistent with the contracts package, where `AgentMode` is:

```ts
["embedded","ophub","sidecar","external"]
```

### Problem

This is a hard protocol mismatch.

Expected v0.3 naming is:

```text
ophub
```

not:

```text
node
```

If OpHub sends `mode: "ophub"` based on contracts, CE rejects it.  
If OpHub sends `mode: "node"` based on CE, contracts and docs drift.

### Required Fix

In CE:

1. Remove the local old-mode schema or align it with contracts.
2. Replace `"node"` with `"ophub"`.
3. Prefer importing and using `registerAgentRequestSchema` from `@xtrape/capsule-contracts-node`.
4. Do not keep separate local v0.3 schema unless there is a strong reason.
5. If `sidecar` and `external` are deployment terms, decide whether they belong in `AgentMode` or only in metadata.

Suggested direction:

```ts
mode: z.enum(["embedded", "ophub"]).default("embedded")
```

or, if keeping broader values:

```ts
mode: z.enum(["embedded", "ophub", "external"]).default("embedded")
```

But avoid `node`.

### Acceptance Criteria

- [ ] CE accepts `mode: "ophub"`.
- [ ] CE no longer accepts or emits `mode: "node"` for new v0.3 runtime behavior.
- [ ] CE and contracts use the same `AgentMode`.
- [ ] CE tests include OpHub registration.
- [ ] Existing `embedded` registration still works.

## BLOCKER-2: Contracts v0.3 still treats sidecar as AgentMode

### Repository

```text
xtrape-capsule-contracts-node
```

### File

```text
src/index.ts
```

### Evidence

Current contract:

```ts
export const AgentMode = ["embedded","ophub","sidecar","external"] as const;
```

### Problem

The latest architecture decision says:

```text
sidecar mode / base-image mode = OpHub deployment forms
not separate Agent modes
```

So `sidecar` should not necessarily be a first-class `AgentMode`.

Recommended meaning:

```text
AgentMode:
  embedded
  ophub

OpHub deployment mode:
  service
  sidecar
  docker
  base-image
```

### Required Fix

Decide and implement one of these approaches:

#### Option A — Strict v0.3

```ts
AgentMode = ["embedded", "ophub"]
```

Add a separate optional field for deployment form only if needed:

```ts
OpHubDeploymentMode = ["service", "sidecar", "docker", "base-image"]
```

#### Option B — Transitional but documented

Keep `external`, but remove `sidecar` from `AgentMode`:

```ts
AgentMode = ["embedded", "ophub", "external"]
```

Then use `metadata.deploymentMode` or a future OpHub-specific schema for sidecar/base-image.

### Acceptance Criteria

- [ ] `AgentMode` no longer uses `node`.
- [ ] `sidecar` is not confused with an Agent mode.
- [ ] README documents Embedded Agent vs OpHub-managed mode.
- [ ] Tests cover `embedded` and `ophub`.
- [ ] If `external` remains, its meaning is documented.

### Problem

Expected demo behavior is:

```text

```

The demo should not create a fake internal `CapsuleNodeAgent` profile. It should expose local `/capsule/*` adapter endpoints and allow actual `xtrape-capsule-ophub-go` to manage it.

### Required Fix

Rename and reshape the demo v0.3 implementation.

#### Replace terminology

Replace:

```text
Node Agent
DEMO_AGENT_MODE=node
CapsuleNodeAgent
node-agent
v0.3-node-agent-events-and-capabilities.md
```

with:

```text
OpHub-managed mode
DEMO_AGENT_MODE=adapter
or DEMO_MODE=adapter
OpHub
ophub-managed
docs/v0.3-ophub-managed-demo.md
```

#### Add actual local adapter endpoints

The demo should expose local HTTP endpoints:

```text
GET  /capsule/manifest
GET  /capsule/health
GET  /capsule/configs
GET  /capsule/actions
POST /capsule/actions/:name/prepare
POST /capsule/actions/:name/execute
```

#### Preserve embedded mode

Keep:

```text
DEMO_AGENT_MODE=embedded
```

or equivalent.

#### Remove fake multi-service “node profile” unless it is clearly replaced by real OpHub-managed behavior

The demo may include a sidekick child service only if it is exposed as a second local adapter service for OpHub to discover/report.

### Acceptance Criteria

- [ ] README no longer says `v0.3 Node Agent`.
- [ ] No new v0.3 doc file uses `node-agent` as the main concept.
- [ ] Demo supports Embedded Agent mode.
- [ ] Demo supports OpHub-managed adapter mode.
- [ ] Adapter mode does not connect directly to Opstage.
- [ ] Adapter mode does not require `@xtrape/capsule-agent-node`.
- [ ] Adapter mode can be managed by actual `xtrape-capsule-ophub-go`.
- [ ] Demo manifest uses `agentMode: "embedded"` for embedded mode and appropriate OpHub-managed metadata only through OpHub-managed flow.
