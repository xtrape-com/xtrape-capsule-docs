## 2. Authoritative v0.3 Model

The expected v0.3 theme is:

```text
OpHub Runtime, Capsule Events and Capability Metadata
```

Terminology rules:

```text
agent-node
  = Node.js Embedded Agent SDK.
  = Keep this repository/package name.

node agent / Node Agent
  = old name for node-level runtime.
  = Replace with OpHub.

OpHub
  = node-level local operations hub runtime.
  = Connects to Opstage and manages one or more local Capsule Services.

OpHub-managed mode
  = Capsule Service does not connect directly to Opstage.
  = Capsule Service exposes local adapter endpoints.
  = OpHub discovers, reports, routes, and forwards results for it.

OpHub Self Service
  = OpHub reports itself as an infrastructure/system Capsule Service.
  = Implemented through a built-in adapter inside OpHub.
```

Expected architecture:

```text
Embedded Agent mode:
  Opstage CE <----polling---- Capsule Service
                                embeds agent-*

OpHub-managed mode:
  Opstage CE <----polling---- OpHub <----local adapter---- Capsule Service A
                                   <----local adapter---- Capsule Service B

OpHub Self Service:
  OpHub reports itself as an infrastructure/system service through a built-in adapter.
```

```text
v0.3.0 branch readiness: Not ready
```

4. Public/internal docs still contain `Node Agent` as the v0.3 roadmap term.
5. `agent-node` still adds an empty `src/node-agent.ts`, which is misleading.

## BLOCKER-1: Demo does not implement `/capsule/*` adapter endpoints

### Repository

```text
xtrape-capsule-demo
```

### Branch

```text
v0.3.0
```

### Files

```text
README.md
src/index.ts
src/server.ts
```

### Evidence

README says the demo exposes these endpoints in OpHub-managed Adapter Mode:

```text
GET  /capsule/manifest
GET  /capsule/health
GET  /capsule/configs
GET  /capsule/actions
POST /capsule/actions/:name/prepare
POST /capsule/actions/:name/execute
```

`src/index.ts` now distinguishes:

```ts
const demoMode = (process.env.DEMO_MODE || "embedded") as "embedded" | "adapter";
```

That is correct.

But `src/server.ts` currently only implements:

```text
GET  /health
GET  /state
POST /state/health-mode
```

There are no `/capsule/*` adapter endpoints.

### Why this blocks v0.3

`OpHub-managed mode` is the core v0.3 demo validation path. Without `/capsule/*`, OpHub cannot discover or manage the demo service.

### Required fix

Implement the local adapter endpoints in `src/server.ts` or a new adapter module.

Required endpoints:

```text
GET  /capsule/manifest
GET  /capsule/health
GET  /capsule/configs
GET  /capsule/actions
POST /capsule/actions/:name/prepare
POST /capsule/actions/:name/execute
```

The implementation should reuse existing demo logic:

```text
createManifest()
demoServiceConfigFromEnv()
getDemoHealth()
getDemoConfigs()
createDemoActions()
```

Expected behavior:

- Adapter mode must not register with Opstage.
- Adapter mode must not require `OPSTAGE_BACKEND_URL`.
- Adapter mode must not require `OPSTAGE_REGISTRATION_TOKEN`.
- Adapter mode must be usable by `xtrape-capsule-ophub-go`.

### Acceptance criteria

- [ ] `GET /capsule/manifest` returns a valid Capsule manifest.
- [ ] `GET /capsule/health` returns current demo health.
- [ ] `GET /capsule/configs` returns demo configs.
- [ ] `GET /capsule/actions` returns demo action catalog.
- [ ] `POST /capsule/actions/:name/prepare` executes the corresponding action prepare path.
- [ ] `POST /capsule/actions/:name/execute` executes the corresponding action execute path.
- [ ] `DEMO_MODE=adapter pnpm dev` starts adapter mode without connecting to Opstage.
- [ ] `DEMO_MODE=embedded pnpm dev` still starts embedded mode.
- [ ] README matches actual behavior.
- [ ] `pnpm typecheck` and `pnpm build` pass.

## BLOCKER-3: CE and contracts still have an `AgentMode` mismatch

### Repositories

```text
xtrape-capsule-contracts-node
xtrape-capsule-ce
```

### Evidence

`contracts-node` defines:

```ts
export const AgentMode = ["embedded","ophub"] as const;
```

CE local schema defines:

```ts
mode: z.enum(["embedded", "ophub", "external"]).default("embedded")
```

### Why this blocks v0.3

CE currently accepts a mode that the public contract does not define. This creates a wire-contract mismatch.

### Required fix

Choose one of these options.

#### Option A: Strict v0.3

CE should match contracts:

```ts
mode: z.enum(["embedded", "ophub"]).default("embedded")
```

#### Option B: Keep `external`

Add `external` to contracts too:

```ts
export const AgentMode = ["embedded", "ophub", "external"] as const;
```

Then document its meaning clearly.

### Strong recommendation

Use Option A unless there is a concrete v0.3 implementation for `external`.

### Additional recommendation

CE should import the register schema from contracts instead of maintaining a divergent local schema:

```ts
import { registerAgentRequestSchema } from "@xtrape/capsule-contracts-node";
```

### Acceptance criteria

- [ ] CE and contracts have exactly the same AgentMode values.
- [ ] `node` is not accepted as a v0.3 mode.
- [ ] `sidecar` is not an AgentMode value.
- [ ] CE tests cover `mode: "ophub"`.
- [ ] CE tests reject unsupported mode values.

## MAJOR-1: OpHub Self Service still lacks explicit wire-level classification

### Repository

```text
xtrape-capsule-contracts-node
```

### Observation

`contracts-node` now supports `embedded | ophub`, capabilities, events, manifests, and multi-service reports.

However, there is no explicit field for:

```text
business service
infrastructure service
system service
```

This makes it harder for CE/UI to distinguish:

```text
OpHub Self Service
OpHub child service
ordinary business service
```

### Recommended fix

Add either:

```ts
export const ServiceKind = ["business", "infrastructure", "system"] as const;
```

and include it in `ReportedServiceSchema`, or document a metadata convention:

```json
{
  "xtrape": {
    "serviceKind": "infrastructure",
    "managedBy": "ophub"
  }
}
```

### Acceptance criteria

- [ ] OpHub Self Service can be represented explicitly or by documented metadata convention.
- [ ] CE can display it differently from child business services.
- [ ] Demo/OpHub examples use the chosen convention.

---

## MAJOR-2: Demo manifest sets `agentMode: "embedded"` even in adapter mode

### Repository

```text
xtrape-capsule-demo
```

### Observation

`createManifest()` currently forces:

```ts
const agentMode: AgentMode = "embedded";
```

The comment says adapter mode still reports as embedded because OpHub manages it.

### Concern

This may be acceptable if `agentMode` means “how this service would integrate if direct”, but it may confuse the model. In OpHub-managed mode, the service is not actually embedded-agent connected.

### Recommended fix

Clarify the contract meaning.

Preferred options:

#### Option A

Remove `agentMode` from child service manifests and let OpHub report the runtime mode.

#### Option B

Use service metadata:

```json
{
  "xtrape": {
    "managedBy": "ophub",
    "adapterProtocol": "http"
  }
}
```

#### Option C

Allow `agentMode: "ophub"` in the manifest when served through adapter mode.

Choose one and keep CE/contracts/demo/OpHub consistent.
