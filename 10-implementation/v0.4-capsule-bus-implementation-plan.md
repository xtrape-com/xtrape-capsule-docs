# v0.4 Implementation Plan — Capsule Bus Experimental

## Overview

This document outlines the implementation plan for **v0.4 Capsule Bus
Experimental**, which introduces governed event-to-command routing
inside a single Opstage CE instance for controlled service
coordination.

**Status:** Planning Phase
**Target Version:** v0.4.0
**Prerequisite:** v0.3 (OpHub Runtime + Capsule Events + Capability
Metadata) must be **tagged, published, and accepted** — see §"v0.3
Prerequisite Checklist" below before starting Phase 1.

The operator-facing concept page is
[`04-opstage/v0.4-capsule-bus-experimental.md`](../04-opstage/v0.4-capsule-bus-experimental.md);
field names, URL paths, and audit event names below are the
canonical record and must match that page.

- [ ] Capability metadata strings (`llm.chat`, `health.probe`, etc.)

- [ ] CE v0.3 audit event names are stable — bus audit events extend
      that namespace.

## Scope

### What's Included in v0.4

1. **Governed event-to-command routing**
   - `BusRoutingRule` schema (Zod, in contracts) with create / list /
     get / update / delete CRUD.
   - Match on `eventType` plus optional `sourceServiceCode`.
   - Per matched route, create **one** `ACTION_EXECUTE` command for
     the target service/action.
   - `DRY_RUN` route status — audit the match without creating a
     command. Useful for staging a new rule before activating it.

2. **Event ingestion**
   - Agent-authenticated endpoint
     `POST /api/agents/{agentId}/bus/events`. The agent's existing
     token determines `sourceServiceCode`; clients cannot forge it.
   - CE stores the event for diagnostics + audit.

3. **Loop / storm safeguards**
   - **Hard max routing depth = 1.** A command's own completion
     **must not** produce a new bus event that re-enters the router.
     This is enforced programmatically: bus events created by a
     command-completion handler are rejected with `BUS_DEPTH_EXCEEDED`.
   - **Per-agent ingestion rate limit** — default `60 events/min`,
     configurable via `OPSTAGE_CAPSULE_BUS_INGEST_PER_MIN`.
   - **Feature flag** — entire bus subsystem gated by
     `OPSTAGE_CAPSULE_BUS_ENABLED=true` (default `false`). When
     disabled, route CRUD returns 503 and the event endpoint
     returns 404.

4. **Operator visibility**
   - Admin UI page for route management (CRUD, enable/disable,
     toggle `DRY_RUN`).
   - Bus audit log view.
   - Audit events: `bus.event.received`, `bus.route.matched`,
     `bus.command.created`, `bus.route.failed`, `bus.route.unmatched`.

5. **Self-routing demo scenario**
   - `demo-worker` emits `demo.item.created`.
   - One route fires `demo-worker.notify` action.
   - Start the demo with route status `DRY_RUN`; verify audit and
     event records; promote to enabled for end-to-end test.

### What's NOT Included in v0.4

- Fan-out: each matched route creates exactly **one** command. The
  `maxCommandsPerEvent` field is **not** in the v0.4 schema; it is
  reserved for v0.6+ alongside any real fan-out implementation.
- Public marketplace or registry workflows.
- Service mesh, mTLS everywhere.
- Complex routing algorithms.
- Load balancing.
- External message broker requirements (Redis, RabbitMQ, NATS,
  Kafka).
- Queue infrastructure requirements.
- Full Capsule Bus runtime (planned for v0.5+ promotion).
- Non-Node OpHub runtime — Go / other runtime hosting is **not** in
  v0.4. Earlier drafts included a Phase 5 "OpHub Go Runtime"; this
  is removed and pushed to v0.5+ when the routing contract has
  proven stable enough to be worth a second-runtime implementation.

## Architecture

### Core flow

```text
┌─────────────────┐
│  Event Source   │ ─── POST /api/agents/{agentId}/bus/events
│  (Capsule       │     (agent-authenticated; sourceServiceCode
│   Service)      │      taken from agent identity, not body)
└────────┬────────┘
         │
         ▼
┌──────────────────┐
│  CE Bus router   │ ─── evaluates enabled routes for
│  (in-process)    │     (eventType, sourceServiceCode)
└────────┬─────────┘
         │
         ├── audit bus.event.received
         ├── audit bus.route.matched (per route)
         │
         ▼
┌─────────────────┐
│  Per matched    │ ─── DRY_RUN → audit bus.route.matched and stop
│  route          │     ENABLED → create one ACTION_EXECUTE command
└────────┬────────┘     and audit bus.command.created
         │
         ▼
┌─────────────────┐
│ Target service  │ ─── receives command via existing
│  (Capsule       │     /api/agents/{agentId}/commands polling
│   Service)      │     loop; reports result as today
└─────────────────┘
```

### Data model — schemas in `xtrape-capsule-contracts-node`

#### 1. `BusRoutingRule`

```ts
import { z } from "zod";

export const BusRoutingRuleStatus = z.enum(["DISABLED", "DRY_RUN", "ENABLED"]);

export const BusRoutingRule = z.object({
  id: z.string().startsWith("brr_"),
  workspaceId: z.string().startsWith("wks_"),
  name: z.string().min(1).max(100),
  description: z.string().max(500).optional(),
  status: BusRoutingRuleStatus,

  // Trigger
  trigger: z.object({
    eventType: z.string().min(1).max(120),
    sourceServiceCode: z.string().max(120).optional(),
  }),

  // Target action — the v0.4 router emits ACTION_EXECUTE
  // commands only. No other commandType is supported.
  action: z.object({
    targetServiceCode: z.string().min(1).max(120),
    actionName: z.string().min(1).max(120),
    payloadTemplate: z.record(z.unknown()).optional(),
  }),

  // Audit
  createdAt: z.string(),
  updatedAt: z.string(),
  createdBy: z.string().startsWith("usr_"),
});
```

Notes:
- Field name is `sourceServiceCode` (not `sourceService`) to match
  the existing CE column naming convention.
- `commandType` from earlier drafts is dropped — the v0.4 router
  only produces `ACTION_EXECUTE`.
- `permissions` block from earlier drafts is dropped — admin-only
  CRUD + agent-authenticated ingestion is enough authorisation
  surface for v0.4.

#### 2. `BusEventEnvelope`

```ts
export const BusEventEnvelope = z.object({
  eventId: z.string().startsWith("bev_"),
  eventType: z.string().min(1).max(120),
  sourceServiceCode: z.string().min(1).max(120),  // server-filled from agent identity
  timestamp: z.string(),                          // ISO 8601
  data: z.unknown(),
  metadata: z
    .object({
      correlationId: z.string().optional(),
      causationId: z.string().optional(),
    })
    .catchall(z.unknown())
    .optional(),
});
```

The agent **must not** set `sourceServiceCode` itself — the CE
ingest endpoint overwrites it from the authenticated agent identity
before persistence, so clients cannot forge source identity.

#### 3. `BusCommandRequest` (internal, not on the wire)

The router does not introduce a new "bus command" type on the wire.
Each matched route creates a normal `ACTION_EXECUTE` command in the
`commands` table, with an additional `sourceEventId` foreign key
pointing at the originating bus event for traceability.

## API surface

All admin routes are gated by the `OPSTAGE_CAPSULE_BUS_ENABLED`
feature flag — when disabled they return `503 Service Unavailable`
with code `BUS_DISABLED`.

```
POST   /api/admin/bus/routes        — Create routing rule
GET    /api/admin/bus/routes        — List routing rules
GET    /api/admin/bus/routes/:id    — Get one routing rule
PATCH  /api/admin/bus/routes/:id    — Update one routing rule
DELETE /api/admin/bus/routes/:id    — Delete one routing rule
GET    /api/admin/bus/events        — List recent bus events (diagnostics)
GET    /api/admin/bus/audit         — Bus-specific audit log filter

POST   /api/agents/{agentId}/bus/events   — Agent-authenticated ingestion
```

`/api/agents/.../bus/events` lives under the existing `/api/agents`
namespace so it uses the same agent-token middleware as
heartbeat / command-result reporting. **No new authentication
scheme** is introduced.

## Feature flag

| Env var | Default | Effect when off |
| --- | --- | --- |
| `OPSTAGE_CAPSULE_BUS_ENABLED` | `false` | Admin route CRUD returns 503; agent event endpoint returns 404; UI page hidden. |
| `OPSTAGE_CAPSULE_BUS_INGEST_PER_MIN` | `60` | Per-agent rate limit on bus event ingestion. Excess returns 429. |

## Security considerations

1. **Service identity verification.** `sourceServiceCode` on
   `BusEventEnvelope` is server-set from the agent token; clients
   cannot forge it. Route `sourceServiceCode` filtering is
   therefore trustworthy.
2. **Default-deny routing.** Events without a matching enabled route
   audit as `bus.route.unmatched` and create no command. There is no
   "catch-all" rule.
3. **Bounded fan-out.** No fan-out in v0.4 — one matched route
   produces exactly one command.
4. **Loop prevention.** Hard `max routing depth = 1`. A command
   created by the router cannot transitively cause another bus
   event to be routed.
5. **Rate limit.** Per-agent ingestion cap rejects with 429
   (`BUS_RATE_LIMITED`); the rejection itself is audited.
6. **Feature flag.** Disabled by default. Operator must explicitly
   opt-in via `OPSTAGE_CAPSULE_BUS_ENABLED=true`.

## Testing strategy

### Unit tests
- Schema validation for `BusRoutingRule` + `BusEventEnvelope`.
- Routing-rule evaluation (match, no-match, dry-run).
- Rate-limit window math.

### Integration tests
- `OPSTAGE_CAPSULE_BUS_ENABLED=false` → admin endpoints 503, agent
  endpoint 404.
- DRY_RUN audit emitted, no command created.
- ENABLED route creates exactly one `ACTION_EXECUTE` command.
- Depth exceeded → 422 `BUS_DEPTH_EXCEEDED`, audit emitted.
- Rate limit → 429 `BUS_RATE_LIMITED`, audit emitted.

### End-to-end tests
- Demo runbook (Phase 4) against a fresh CE container.

### Performance tests
- Event throughput at 60 events/min/agent across N agents.
- Routing-rule eval latency vs. rule count.

## Success criteria

v0.4 is considered successful when:

1. ✅ Routes can be created, updated, deleted via the admin API.
2. ✅ Agents can publish events via `publishEvent`.
3. ✅ Routing engine evaluates rules and creates commands.
4. ✅ Commands are delivered to target services through the existing
       command polling loop.
5. ✅ Every routing decision is audited.
6. ✅ UI allows operators to manage routes and view event flow.
7. ✅ Demo runbook works end-to-end against a freshly built CE
       container.
8. ✅ Feature flag controls enable/disable globally.
9. ✅ `max routing depth = 1` enforced programmatically.
10. ✅ Per-agent rate limit enforced.
11. ✅ Documentation (concept page + tutorial + reference + release
        notes) is complete.
12. ✅ Security review passes.

## Risks and mitigations

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Scope creep into v0.5 catalog work | High | Keep prototype minimal; `maxCommandsPerEvent` / capability-based routing / public registry are **explicitly** v0.5+ |
| Routing loops | High | `max routing depth = 1` hard-enforced server-side; not a documentation rule |
| Event storm | Medium | Per-agent rate limit (60/min default); operator-tunable |
| Source identity spoofing | High | `sourceServiceCode` server-set from agent token, never read from body |
| Performance bottleneck on rule eval | Medium | Bound rule count via admin UI warning at > 100 rules; profile before release |
| 0.4 SDK against 0.3 CE | Medium | `publishEvent` surfaces `BusDisabledError` cleanly; operator handles at startup |
| Insufficient testing | Medium | Demo runbook + integration matrix + perf bench before tag |
| Documentation gaps | Low | Concept page already drafted; reference + tutorial written alongside Phase 5 |
