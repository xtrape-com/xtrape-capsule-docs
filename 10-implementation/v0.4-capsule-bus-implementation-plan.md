# v0.4 Implementation Plan — Capsule Bus Experimental

## Overview

This document outlines the implementation plan for **v0.4 Capsule Bus Experimental**, which introduces governed event-to-command routing for controlled service coordination.

**Status:** Planning Phase  
**Target Version:** v0.4.0  
**Prerequisite:** v0.3 (Capsule Events and Capability Metadata) must be completed first

## Scope

### What's Included in v0.4

1. **Governed Event-to-Command Routing**
   - Define routing rules that transform events into commands
   - Operator-visible routing configuration
   - Audit trail for routing decisions

2. **Service Coordination Foundations**
   - Service identity verification
   - Permission-based routing
   - Guardrails for service-to-service communication

3. **Experimental Prototype**
   - In-process or local bus implementation
   - Disabled by default
   - Feature-flag controlled activation

4. **Operator Visibility**
   - Routing rule management UI
   - Event/command flow visualization
   - Audit logs for bus operations

### What's NOT Included in v0.4

- Public marketplace or registry workflows
- Service mesh or mTLS everywhere
- Complex routing algorithms
- Load balancing
- External message broker requirements (Redis, RabbitMQ, NATS, Kafka)
- Queue infrastructure requirements
- Full Capsule Bus runtime (planned for later versions)

## Architecture

### Core Components

```text
┌─────────────────┐
│  Event Source   │ ─── publishes event
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Capsule Bus    │ ─── evaluates routing rules
│  (Experimental) │     checks permissions
└────────┬────────┘     creates command
         │
         ▼
┌─────────────────┐
│ Command Target  │ ─── receives command
│  (Service B)    │     executes action
└─────────────────┘
```

### Data Model Extensions

#### 1. Bus Routing Rule

```typescript
interface BusRoutingRule {
  id: string;
  name: string;
  description?: string;
  enabled: boolean;
  
  // Trigger condition
  trigger: {
    eventType: string;      // e.g., "service.health.degraded"
    sourceService?: string; // optional filter
    metadata?: Record<string, any>; // additional filters
  };
  
  // Action to take
  action: {
    targetType: 'command';
    targetService: string;
    commandType: string;    // e.g., "restart_service"
    payload?: Record<string, any>;
  };
  
  // Governance
  permissions: {
    requiredRole?: string;
    allowedSources?: string[];
    allowedTargets?: string[];
  };
  
  // Audit
  createdAt: string;
  updatedAt: string;
  createdBy: string;
}
```

#### 2. Bus Event Envelope

```typescript
interface BusEventEnvelope {
  eventId: string;
  eventType: string;
  sourceService: string;
  timestamp: string;
  data: any;
  metadata?: {
    correlationId?: string;
    causationId?: string;
    [key: string]: any;
  };
}
```

#### 3. Bus Command Request

```typescript
interface BusCommandRequest {
  commandId: string;
  commandType: string;
  targetService: string;
  sourceEventId?: string; // link to triggering event
  payload?: any;
  metadata?: {
    routingRuleId?: string;
    correlationId?: string;
    [key: string]: any;
  };
}
```

## Security Considerations

1. **Permission Model**
   - Routing rules require explicit permissions
   - Service identity verification mandatory
   - No implicit trust between services

2. **Audit Requirements**
   - All routing decisions logged
   - Event ingestion audited
   - Command creation traced to source event
   - Rule changes tracked

3. **Guardrails**
   - Rate limiting on event ingestion
   - Maximum routing depth (prevent loops)
   - Circuit breaker for failed targets
   - Default-deny routing policy

4. **Feature Flag**
   - Bus disabled by default
   - Requires explicit enablement
   - Can be disabled per-environment

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Complexity exceeds v0.4 scope | High | Keep prototype minimal, defer advanced features |
| Performance bottlenecks | Medium | Profile early, optimize routing engine |
| Security vulnerabilities | High | Security review before release, default-deny policy |
| Insufficient testing | Medium | Comprehensive test suite, E2E scenarios |
| Documentation gaps | Low | Start docs early, iterate with implementation |
