# v0.4 Capsule Bus Experimental

Capsule Bus v0.4 is an experimental, default-disabled, Opstage-governed event-to-command routing foundation.

## Accepted surface

- Capsule Services or OpHub publish event envelopes to CE.
- CE stores the event for diagnostics and auditability.
- Enabled route rules match `eventType` plus optional `sourceServiceCode`.
- A matched route may create an `ACTION_EXECUTE` command for a target service/action.
- `DRY_RUN` routes audit matches without creating commands.

The richer Capsule Catalog work (Service Manifest v1, endpoint visibility, capability metadata, discovery-by-capability) belongs to **v0.5** — see `07-roadmap/00-version-roadmap.md` §13. v0.4 ships before the catalog on purpose, so the catalog can be designed against real routing traffic.

## Non-goals

Capsule Bus v0.4 is not a workflow engine, service mesh, distributed transaction layer, or general-purpose message broker replacement. Explicit non-goals in this cut:

- **Fan-out.** One matched route produces one command. The `maxCommandsPerEvent` concept from earlier drafts has been removed from the v0.4 schema entirely; it is reserved for v0.6+.
- **Public registry / marketplace workflows** (v0.6 / v0.7).
- **External message brokers** (Redis, RabbitMQ, NATS, Kafka).
- **mTLS / service mesh.**
- **Non-Node OpHub runtimes** — Go and other runtime hosting are v0.5+ work.

## Loop and storm safeguards

- Routes are **disabled by default**.
- **`DRY_RUN` mode** is supported for staging validation before enabling.
- **Max routing depth = 1**, programmatically enforced. A command produced by the router cannot transitively cause another bus event to be routed. The CE ingestion endpoint rejects events whose `causationId` points at a router-created command with error code `BUS_DEPTH_EXCEEDED`. This replaces the earlier "operators should avoid loops" honor-system note with a hard backend constraint.
- **Per-agent ingestion rate limit** — default `60 events/min`, configurable via `OPSTAGE_CAPSULE_BUS_INGEST_PER_MIN`. Excess returns 429 (`BUS_RATE_LIMITED`); the rejection is itself audited.

## Audit events

Every routing decision produces exactly one event from this set:

| Audit action | When | Companion event |
| --- | --- | --- |
| `bus.event.received` | Always, on every successful ingestion. | — |
| `bus.route.matched` | Per matched enabled route (any status). | If route is `ENABLED` → followed by `bus.command.created` for the same matched-route id. If route is `DRY_RUN` → not followed by anything else. |
| `bus.command.created` | A command was created from a matched, enabled route. | Preceded by `bus.route.matched` for the same route. |
| `bus.route.failed` | Route matched but command creation failed (target service missing, action not registered, agent offline at dispatch time, etc.). | Preceded by `bus.route.matched`. |
| `bus.route.unmatched` | Event ingested but no enabled route matched. | — |

`bus.route.matched` always appears alone in DRY_RUN mode and paired with either `bus.command.created` or `bus.route.failed` in ENABLED mode. Operators can use this pattern to spot routes that match but never produce commands.

## Prototype scenario

`demo-worker` emits `demo.item.created` whenever its `createDemoItem` action runs. A CE route targets `demo-worker.notify` on the same service. Start with route status `DRY_RUN`, run `createDemoItem`, and verify `bus.event.received` + `bus.route.matched` audit rows appear with no command. Then flip the route to `ENABLED`, re-run, and verify `bus.command.created` followed by the `notify` action executing.

The runnable version of this scenario lives at `xtrape-capsule-demo/docs/v0.4-bus-smoke-test.md` (added in demo Phase 4 work).

## Implementation plan alignment

The detailed planning reference is `10-implementation/v0.4-capsule-bus-implementation-plan.md`. Field names, URL paths, audit events, and error codes on this page are the canonical record — the implementation plan defers to it. Anchor decisions:

- **Admin route management** uses `/api/admin/bus/routes` (not a new `/api/v1/...` namespace).
- **Agent event ingestion** uses `POST /api/agents/{agentId}/bus/events`, so source identity is tied to the existing agent token model. CE overwrites `sourceServiceCode` from the authenticated agent identity; clients cannot forge it.
- **Feature flag**: `OPSTAGE_CAPSULE_BUS_ENABLED=true` (default `false`) gates the entire subsystem.
- **Field name**: `sourceServiceCode` (not `sourceService`).
- **Command type**: routes emit `ACTION_EXECUTE` only; no new wire command type.
- **`maxCommandsPerEvent`**: removed from v0.4 schema (was half-implemented in earlier drafts).

## v0.3 → v0.4 SDK back-compat

The Agent SDK gains `publishEvent(...)` in 0.4.0. When a 0.4.0 SDK runs against a 0.3.x CE backend, the missing `/api/agents/.../bus/events` endpoint returns 404; the SDK surfaces `BusDisabledError`. Callers should catch this once at startup and stop calling `publishEvent` — it is not a transient failure, and the SDK's standard retry/backoff does **not** kick in.
