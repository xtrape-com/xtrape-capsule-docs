## Decision

The word "contracts" in this project refers to **two distinct layers** that MUST NOT be conflated:

1. **Layer 1 — Spec** (machine-readable, language-agnostic, human-authored)
2. **Layer 2 — Bindings** (per-language code, generated from Layer 1)

CE v0.1 chooses **Path C** of the contracts strategy: Layer 1 lives in the `xtrape-capsule-docs` repository under
`09-contracts/`; Layer 2 lives in **per-language repositories** (`xtrape-capsule-contracts-node`, future `-python`,
`-java`, `-go`).

This ADR defines what each layer contains, where it lives, how Layer 2 is generated and validated, and how the layers stay synchronized.

This ADR is normative. Older documents that conflate "contracts" with a single TS package MUST defer to this ADR.

## Layer 1 — Spec (lives in `xtrape-capsule-docs/09-contracts/`)

Layer 1 is the **single source of truth** for all wire-level shapes, codes, names, and constants. It is hand-written and version-controlled in the docs repo.

### Layer 1 contents

```text
xtrape-capsule-docs/09-contracts/
├── README.md                          (navigation; references this ADR)
├── openapi/                           ← OpenAPI wire-format SSOT
│   ├── opstage-ce-v0.1.yaml           (primary spec file)
│   └── redocly.yaml                   (lint config)
├── prisma/                            ← CE persistence schema (review copy;
│   │                                    authoritative copy lives in xtrape-capsule-ce/packages/db/)
│   ├── schema.prisma
│   └── prisma.config.ts
├── enums/                             ← non-OpenAPI machine-readable SSOTs
│   ├── status-enums.json              (AgentStatus, CapsuleServiceStatus, CommandStatus, etc.)
│   ├── audit-actions.json             (canonical AuditEvent.action values)
│   └── id-prefixes.json               (newId(prefix) prefix table)
├── errors.json                        ← canonical error codes + HTTP statuses (SSOT)
├── errors.md                          ← rendered from errors.json (human reading; CI checks parity)
├── examples/                          ← wire-format fixtures used by every binding's conformance test
│   ├── command-action-pending.json
│   ├── audit-event-service-action-requested.json
│   ├── register-agent-request.json
│   └── ...                            (one fixture per representative wire shape)
└── tools/                             ← markdown renderers (no language bindings here)
    └── render-errors.ts
```

The "Layer 1 SSOT" set, in priority order for downstream consumers:

1. `openapi/opstage-ce-v0.1.yaml` — the wire format.
2. `errors.json` — error codes.
3. `enums/*.json` — shared enumerations.
4. `examples/*.json` — fixtures that every binding MUST round-trip.

`prisma/schema.prisma` is included for documentation/review only; the **authoritative** copy is `xtrape-capsule-ce/packages/db/schema.prisma` (CE-bound, not edition-agnostic).

### Layer 1 rules

- **Edit only files under `spec/`** when changing wire format. Other files (`errors.md`, generated tables) are derived.
- All `spec/*.json` files MUST validate against a JSON Schema that lives next to them (e.g. `spec/errors.schema.json`). The schema acts as a guardrail for hand-edits.
- The OpenAPI YAML MUST lint clean against `redocly.yaml`. Any new warning MUST be either fixed or explicitly suppressed in `redocly.yaml` with a comment.
- `spec/examples/*.json` MUST round-trip through every Layer 2 binding's parser without loss. CI in each binding repo enforces this (see "Conformance Tests" below).
- `errors.md` MUST be regenerated by `tools/render-errors.ts` and committed; CI fails the PR if `errors.md` and `errors.json` disagree.

### Forbidden in Layer 1

- ❌ Language-specific code (`.ts`, `.py`, `.java`, etc.)
- ❌ Edition-specific endpoints — `09-contracts/spec/openapi/opstage-ee-v*.yaml` (future) is allowed; `opstage-ce-v0.1.yaml` MUST NOT contain EE/Cloud-only endpoints.
- ❌ Implementation hints unrelated to the wire format (those belong to `08-decisions/` ADRs or `10-implementation/`).

## Layer 2 — Bindings (one repository per language)

Layer 2 packages are **published runtime libraries** that consumers import. Each language has its own repo, its own native toolchain, its own version, and its own release cadence.

### Layer 2 repositories (CE v0.1 + planned)

| Repo | Status | Package | Toolchain |
|---|---|---|---|
| `xtrape-capsule-contracts-node` | **CE v0.1** | `@xtrape/capsule-contracts-node` (npm) | pnpm + tsup + openapi-typescript + Zod |
| `xtrape-capsule-contracts-python` | future | `xtrape-capsule-contracts-python` (PyPI) | uv/poetry + datamodel-code-generator + Pydantic v2 |
| `xtrape-capsule-contracts-java` | future | `com.xtrape:capsule-contracts-java` (Maven Central) | Gradle + openapi-generator + Jackson |
| `xtrape-capsule-contracts-go` | future | `github.com/xtrape/xtrape-capsule-contracts-go` | go modules + oapi-codegen |

CE v0.1 only ships the Node binding. The other repos are listed here so future implementers follow the same pattern (and to fix the naming up front).

### Layer 2 repository skeleton (Node example, applicable shape for all languages)

```text
xtrape-capsule-contracts-node/
├── README.md                          (1-paragraph overview + compatibility table)
├── package.json                       (name: @xtrape/capsule-contracts-node)
├── tsconfig.json
├── tsup.config.ts                     (ESM + CJS + .d.ts output)
├── spec/                              ← read-only mirror of xtrape-capsule-docs/09-contracts/
│   ├── .source                        (one line: "xtrape-capsule-docs@<commit-sha>")
│   ├── openapi/opstage-ce-v0.1.yaml
│   ├── errors.json
│   ├── enums/status-enums.json
│   ├── enums/audit-actions.json
│   ├── enums/id-prefixes.json
│   └── examples/...
├── src/
│   ├── index.ts                       (single entry, re-exports everything)
│   ├── types.gen.ts                   (generated; openapi-typescript)
│   ├── errors.gen.ts                  (generated; from errors.json)
│   ├── enums.gen.ts                   (generated; from status-enums.json)
│   ├── audit.gen.ts                   (generated; from audit-actions.json)
│   ├── id.ts                          (hand-written: newId(prefix), uses id-prefixes.json)
│   └── schemas.ts                     (hand-written Zod, mirrors OpenAPI components/schemas)
├── tests/
│   ├── conformance.spec.ts            (parses every spec/examples/*.json with schemas.ts)
│   └── enum.spec.ts                   (asserts every spec status enum is exported)
├── tools/
│   ├── sync-spec.ts                   (pulls spec/ from xtrape-capsule-docs HEAD)
│   └── codegen.ts                     (regenerates src/*.gen.ts)
├── .github/workflows/
│   ├── ci.yml                         (sync-spec --check + codegen --check + test + lint)
│   ├── release.yml                    (changesets → npm publish)
│   └── upstream-bump.yml              (cron: weekly sync from xtrape-capsule-docs)
├── CHANGELOG.md                       (managed by changesets)
├── LICENSE
└── .gitignore
```

### Layer 2 rules (apply to every language repo)

1. **`spec/` is read-only and mirrored**, not authored. PRs MAY NOT edit `spec/` directly. The `tools/sync-spec.ts` script is the only writer.
2. **All `*.gen.*` files are committed** (so consumers can read them on GitHub) but CI runs `codegen --check` to ensure they match the inputs in `spec/`.
3. **A `spec/.source` file** records the commit SHA in `xtrape-capsule-docs` that the current `spec/` copy mirrors. This
is how downstream agent SDK CI proves "I am building against docs commit `abc1234`".
4. **Conformance tests are non-skippable**: every `spec/examples/*.json` MUST parse with the binding's runtime schemas.
If a fixture doesn't parse, the binding repo's CI fails until either the binding is fixed or the fixture is updated
upstream.
5. **No reverse dependencies**: a binding repo MUST NOT import from any agent SDK or any backend repo (enforced via ADR 0008's reverse-dep CI rule).

## Sync Workflow: Layer 1 → Layer 2

This is the most operationally important part of Path C and MUST be implemented exactly as described.

### Step 1: Author edits Layer 1 in docs repo

```bash
git clone github.com/xtrape/xtrape-capsule-docs
# edit 09-contracts/openapi/opstage-ce-v0.1.yaml or any *.json under 09-contracts/{enums,examples}/ or errors.json
# regenerate errors.md (see "Render Tools" below)
git commit -m "feat(spec): add Command.priority field"
# PR review, merge to main
```

### Step 2: Each binding repo's `upstream-bump.yml` workflow runs

A scheduled workflow (`cron: '0 6 * * *'` daily) in each binding repo:

1. Fetches `xtrape-capsule-docs` at HEAD;
2. Runs `pnpm tools/sync-spec.ts --from <docs-clone>/09-contracts`;
3. Runs `pnpm tools/codegen.ts`;
4. Runs `pnpm test` (conformance + unit);
5. If `git status` is dirty, opens a PR titled `chore(spec): bump to xtrape-capsule-docs@<sha>` with the diff;
6. Updates `spec/.source` to the new SHA in the same commit.

If conformance fails, the workflow opens an **issue** instead of a PR, with the failing test output, so a human author
can update the binding's hand-written code (e.g. add a new Zod schema for a new endpoint).

### Step 3: Binding repo human reviews and merges

The PR opened by `upstream-bump.yml` runs the full CI:

- `tools/sync-spec.ts --check` (no diff)
- `tools/codegen.ts --check` (no diff after regen)
- `pnpm test` (conformance + unit + reverse-dep check)
- `pnpm lint`
- `pnpm typecheck`

A human reviewer ensures any new hand-written code (new Zod schema, new helper) is correct, then merges. Changesets
bumps the version (typically `minor` for new fields, `major` for incompatible removals).

### Step 4: Release publishes Layer 2

`release.yml` runs on tag push (driven by changesets). It:

- builds (`tsup`);
- publishes `@xtrape/capsule-contracts-node@X.Y.Z` to npm with provenance (`npm publish --provenance`);
- pushes the tag and updates `CHANGELOG.md`.

### Step 5: Downstream consumers (agent + CE) get the new version

Renovate in `xtrape-capsule-agent-node` and `xtrape-capsule-ce` opens a PR within ~1 hour of the npm publish
(configurable). Each consumer's CI runs against the new bindings; on green, the PR is auto-merged (patch) or reviewed
(minor/major).

## Render Tools (live in the docs repo)

The docs repo MUST contain a tiny `09-contracts/tools/` directory with renderers that turn the JSON SSOTs into
human-readable markdown. CI in `xtrape-capsule-docs` runs them in `--check` mode and fails the PR if a generated file is
stale.

```text
xtrape-capsule-docs/09-contracts/tools/
├── render-errors.ts                   (errors.json           → 09-contracts/errors.md)
├── render-status-enums.ts             (enums/status-enums.json → 02-specs/09-status-model-spec.md sections)
└── render-audit-actions.ts            (enums/audit-actions.json → 02-specs/08-audit-event-spec.md sections)
```

These renderers are hand-written, dependency-free TypeScript scripts (run via `tsx`). They do NOT generate code — only
documentation. They keep the human-readable `*.md` files in sync with the machine-readable JSON.

## Conformance Tests

Every Layer 2 binding repo MUST run conformance tests against `spec/examples/*.json`. The test surface is:

```ts
// tests/conformance.spec.ts (Node binding example)
import { describe, expect, test } from "vitest";
import { readFileSync, readdirSync } from "node:fs";
import * as schemas from "../src/schemas";

const fixtureDir = "spec/examples";
const fixtures = readdirSync(fixtureDir).filter(f => f.endsWith(".json"));

describe("conformance", () => {
  for (const f of fixtures) {
    test(`example ${f} parses`, () => {
      const raw = JSON.parse(readFileSync(`${fixtureDir}/${f}`, "utf8"));
      // each fixture has a top-level "$kind" pointing at the schema name
      const schema = (schemas as any)[raw.$kind];
      expect(schema, `unknown $kind: ${raw.$kind}`).toBeDefined();
      const result = schema.safeParse(raw.payload);
      expect(result.success, JSON.stringify(result.error?.issues)).toBe(true);
    });
  }
});
```

Each fixture file in `spec/examples/` has shape:

```json
{
  "$kind": "Command",
  "$comment": "minimum action command in PENDING state",
  "payload": { /* the actual wire shape */ }
}
```

This `$kind` indirection lets every language binding pick the correct schema by name without per-language coordination.

## Versioning and Compatibility

- Each binding repo versions independently using **semver**:
  - `patch`: spec change is purely additive on the wire and the binding adopts it without API change.
  - `minor`: spec adds new types or fields; binding's public API gains additions.
  - `major`: spec removes/renames a wire field; binding's public API breaks.
- The `xtrape-capsule-docs` repo carries a `09-contracts/spec/openapi/opstage-ce-v0.1.yaml` file whose `info.version` (semver) tracks **the wire compatibility level**, not the binding versions.
- Each binding repo's README MUST contain a Compatibility table:

```markdown
| Binding version | Spec version (info.version) | docs commit |
|-----------------|-----------------------------|-------------|
| 0.1.0           | 0.1.0                       | abc1234     |
| 0.1.1           | 0.1.0                       | def5678     |
| 0.2.0           | 0.2.0                       | 9abcdef     |
```

## Future Migration: Path C → Path A (NOT planned, but reversible)

If we later decide to consolidate per-language bindings into a single polyglot repo, the migration path is:

1. Create new `xtrape-capsule-contracts` repo with `packages/{ts,py,java,go}/` subdirs.
2. `git mv` each `xtrape-capsule-contracts-<lang>` repo's contents under the matching subdir using `git filter-repo --to-subdirectory-filter packages/<lang>` to preserve history.
3. Update Renovate and CI in `xtrape-capsule-agent-*` / `xtrape-capsule-ce` to track the new repo path. **npm/PyPI/Maven package names DO NOT change**, so consumers are unaffected.
4. Archive the old per-language repos with a redirect notice.

This safety hatch is documented to keep Path C reversible.

## Forbidden in CE v0.1

- ❌ Putting Layer 1 SSOT inside any binding repo (binding repos mirror, never originate).
- ❌ Generating Layer 2 at consumer install time (consumers depend on the published, prebuilt npm package).
- ❌ Hand-editing `*.gen.*` files.
- ❌ Skipping conformance tests in CI.
- ❌ Publishing a binding version against a Layer 1 spec that is not committed to `xtrape-capsule-docs` `main`.

## Acceptance Criteria

- `xtrape-capsule-docs/09-contracts/` contains `errors.json` (top-level), `enums/{status-enums,audit-actions,id-prefixes}.json`, and the existing `openapi/` and `prisma/` subdirectories.
- `09-contracts/errors.md` is byte-for-byte the output of `09-contracts/tools/render-errors.ts 09-contracts/errors.json`.
- `xtrape-capsule-contracts-node` repo skeleton exists (even if empty `src/` initially), with `spec/` mirrored from docs and `tools/sync-spec.ts` working.
- `xtrape-capsule-contracts-node` CI runs `--check` versions of sync, codegen, conformance.
- A successful release publishes `@xtrape/capsule-contracts-node@0.1.0` to npm with provenance.
- `xtrape-capsule-agent-node` and `xtrape-capsule-ce` each install `@xtrape/capsule-contracts-node` from npm, never via `link:` or `file:` paths in committed `package.json`.
