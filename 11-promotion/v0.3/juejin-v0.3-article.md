# Docker Compose 之后，小团队服务治理还缺什么？

很多小团队做私有化部署时，第一步通常不是 Kubernetes，而是 Docker Compose。

这很合理。

Docker Compose 简单、直接、可控。对于一台或几台服务器、几个到十几个服务的私有化部署场景来说，它往往比 Kubernetes 更容易落地。

但是，当服务数量逐渐增加后，一个新的问题会出现：

> 服务跑起来了，但服务本身如何治理？

这也是我在做 **xtrape-capsule v0.3** 时重点思考的问题。

## 一、Docker Compose 解决的是“运行”，不是完整的“治理”

Docker Compose 很适合解决这些问题：

- 如何启动服务；
- 如何配置容器；
- 如何声明服务之间的依赖；
- 如何管理环境变量；
- 如何在小规模环境中快速部署。

但它并不完整解决下面这些问题：

- 当前有哪些业务服务已经注册？
- 某个服务运行在哪个节点？
- 当前服务版本是什么？
- 服务是否只是容器存活，还是业务层面也健康？
- 节点和服务之间是什么关系？
- 是否有统一的运维入口？
- 是否需要每个服务都暴露一个管理 API？
- 小团队是否真的应该为了这些问题直接上 Kubernetes？

这些问题不是单纯的容器管理问题，而是服务治理问题。

---

## 二、小团队私有化部署不一定一开始就需要 Kubernetes

Kubernetes 很强，但它的心智成本和运维成本也很高。

对于一些小型私有化部署，实际情况可能是：

- 只有 1-3 台机器；
- 只有 5-20 个服务；
- 没有专职运维团队；
- 部署在客户自己的私有环境；
- 网络访问受限；
- 不希望暴露太多管理端口；
- 更关注稳定运行和可观察，而不是复杂编排。

在这种情况下，直接引入 Kubernetes 不一定是最优解。

很多时候，小团队真正需要的是一个更轻的中间层：

> 在 Kubernetes 之前，先解决轻量服务的注册、状态、版本、节点关系和统一管理问题。

这就是 xtrape-capsule 想尝试的方向。

---

## 三、xtrape-capsule 的定位：轻服务治理

xtrape-capsule 不是为了替代 Docker Compose，也不是为了替代 Kubernetes。

我更倾向于把它定位为：

> 面向小型私有化部署的轻服务治理平台。

它关注的不是“如何调度容器”，而是：

- 如何定义一个可治理的轻服务；
- 如何让服务注册到统一平台；
- 如何查看服务和节点之间的关系；
- 如何在不暴露每个服务管理接口的情况下进行统一治理；
- 如何为小团队提供一个比 Kubernetes 更轻的治理模型。

---

## 四、核心模型：opstage、ophub、capsule-service

当前 v0.3 中，xtrape-capsule 主要围绕三个概念展开。

### 1. opstage

opstage 是统一运维平台。

它负责提供集中管理入口，后续可以承载服务列表、节点关系、状态展示、版本管理、配置管理等能力。

### 2. ophub

ophub 是节点侧的 hub / agent。

它的重点是：由节点侧主动与 opstage 通信，而不是要求 opstage 直接访问每个节点的管理接口。

这种模式更适合私有化部署，因为它可以减少节点暴露的管理端口。

### 3. capsule-service

capsule-service 是被治理的轻服务。

它可以是一个业务服务，也可以是一个较小的后台能力单元。它通过 capsule 模型被注册、观测和管理。

---

## 五、为什么不希望每个服务都暴露管理 API？

在小型私有化部署中，管理入口越多，安全面越复杂。

如果每个服务都暴露自己的管理 API 或管理后台，会带来几个问题：

- 每个入口都需要认证和授权；
- 每个入口都可能配置错误；
- 每个入口都需要网络策略保护；
- 服务越多，攻击面越大；
- 小团队很难长期维护统一安全标准。

因此，xtrape-capsule 更倾向于：

> 服务和节点侧通过 ophub 主动连接中心平台，由 opstage 提供统一管理入口。

这样可以降低私有化环境中的暴露面，也更符合“小团队低成本运维”的目标。

---

## 六、v0.3 的重点不是成熟产品，而是架构验证

xtrape-capsule v0.3 仍然是早期版本。

这次发布的核心目标不是宣称它已经生产可用，而是希望获得 public review，尤其是下面几个问题的反馈：

1. “Kubernetes 之前的轻服务治理层”这个问题是否真实存在？
2. opstage / ophub / capsule-service 这个模型是否容易理解？
3. 节点侧主动连接中心平台的方式是否适合私有化部署？
4. 小团队最小可用的服务治理能力应该包含哪些？
5. xtrape-capsule 和 Docker Compose、Portainer、Kubernetes 的边界应该如何划分？

---

## 七、我目前的判断

在小团队私有化部署场景中，技术选型不应该只有两端：

> 要么 Docker Compose 简单跑起来；  
> 要么直接上 Kubernetes。

中间应该存在一个更轻的治理层。

它不负责复杂编排，不追求大规模集群调度，而是先解决：

- 服务注册；
- 服务状态；
- 版本信息；
- 节点关系；
- 统一管理；
- 安全暴露面控制。

这就是 xtrape-capsule 当前探索的方向。
