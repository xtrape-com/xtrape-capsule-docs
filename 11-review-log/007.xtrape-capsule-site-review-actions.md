# xtrape-capsule-site 修改意见与优化任务清单

## 目标

将 `xtrape-capsule-site` 从“对外文档站骨架”提升为适合 `xtrape-capsule CE v0.1.0 Public Preview` 推广使用的官网与文档入口。

当前站点方向正确，但仍缺少推广级官网所需的关键元素：

- 真实截图 / 架构图 / Demo 视觉材料；
- 明确、可执行的 Quick Start；
- 确定性的 Node Agent SDK 接入示例；
- Public Preview 状态提示；
- GitHub Pages / 自定义域名发布闭环；
- Use Cases 深度内容；
- License / Trademark 边界说明。

本文件用于交给 AI / Code Agent 执行优化。

---

## 总体判断

当前 `xtrape-capsule-site` 已经具备 public-facing site 的基础形态：

- 使用 VitePress；
- 有首页、Quick Start、Concepts、Agents、Contracts、Use Cases、Editions、Security、Roadmap、FAQ 等基本结构；
- 首页定位基本清楚；
- FAQ 已经解释与 Nacos / Apollo / Prometheus / Backstage 的区别；
- Security / Token Model 页面有较好的安全边界说明。

但目前更适合小范围预览，不适合直接作为大范围推广官网。

优化目标：

> 让外部用户在 30 秒内理解项目，在 5 分钟内跑起 Opstage CE，在 10 分钟内接入第一个 Capsule Service。

## 1. 检查并格式化所有 Markdown 文件

### 问题

部分 Markdown 文件可能存在内容被压缩到一行的问题，尤其是：

```text
docs/index.md
docs/.vitepress/config.ts
```

这会影响：

- 后续维护；
- AI 编辑；
- Pull Request review；
- 文档可读性。

### 修改要求

将所有 Markdown 文件整理为标准格式：

- YAML frontmatter 独立成块；
- 标题层级清晰；
- 代码块语言标注准确；
- 段落换行自然；
- 列表缩进统一；
- 避免整篇文档压缩成一行。

### 首页 frontmatter 示例

```md
---
layout: home
hero:
  name: Xtrape Capsule
  text: A lightweight control plane for AI-era Capsule Services.
  tagline: Connect small services, automation workers, integration services, and agent runtimes to a unified Opstage console through an Agent SDK.
  actions:
    - theme: brand
      text: Get Started
      link: /getting-started/quick-start
    - theme: alt
      text: View on GitHub
      link: https://github.com/xtrape-com/xtrape-capsule-ce
features:
  - title: Capsule Services
    details: Self-describing lightweight services designed for AI-era automation and runtime governance.
  - title: Agent Registration
    details: Services connect through registered Agents using outbound communication and token-based authorization.
  - title: Opstage Control Plane
    details: Observe health, expose configs, execute actions, and keep audit trails from a unified console.
---
```

### 验收标准

- `docs/index.md` 在 GitHub raw 和编辑界面中结构清晰；
- VitePress build 正常；
- Markdown lint 基本无明显格式问题；
- 所有主要页面可维护、可继续扩展。

## 2. 确认 VitePress build 与 GitHub Pages 发布闭环

### 问题

当前仓库已有 GitHub Pages workflow，但需要确认站点是否真实可访问。

### 修改要求

检查：

```text
.github/workflows/deploy.yml
docs/.vitepress/config.ts
package.json
```

确保：

- `pnpm install` 正常；
- `pnpm docs:build` 或对应 build 命令正常；
- GitHub Pages workflow 可成功部署；
- 仓库 Settings > Pages 已配置正确；
- GitHub About 的 website 已填入站点 URL。

### 如果使用 GitHub Pages 项目页

当前 base 应为：

```ts
base: "/xtrape-capsule-site/"
```

站点 URL 通常为：

```text
https://xtrape-com.github.io/xtrape-capsule-site/
```

### 如果使用自定义域名

例如：

```text
https://capsule.xtrape.com
```

则 VitePress base 应调整为：

```ts
base: "/"
```

并在 README 中说明：

```text
If deploying under a custom domain, set VitePress base to "/".
```

### 验收标准

- GitHub Pages 可访问；
- 首页、导航、侧边栏、搜索可正常使用；
- 静态资源路径正常；
- README 和仓库 About 均指向正确站点 URL。

## 5. 修正 Quick Start 中 Docker 镜像状态

### 问题

Quick Start 中使用：

```bash
ghcr.io/xtrape-com/xtrape-capsule-ce:latest
```

但如果镜像尚未发布，会导致用户执行失败。

### 修改要求

根据真实情况二选一。

### 情况 A：镜像已经发布

保留一行启动命令：

```bash
docker run -p 8080:8080 -v opstage-data:/app/data ghcr.io/xtrape-com/xtrape-capsule-ce:latest
```

并说明：

```text
Open http://localhost:8080 after the container starts.
```

### 情况 B：镜像尚未发布

不要把 GHCR 镜像作为主路径。改为源码启动：

```bash
git clone https://github.com/xtrape-com/xtrape-capsule-ce.git
cd xtrape-capsule-ce
docker compose up --build
```

并加说明：

```text
Public Docker images are planned for the v0.1.0 Public Preview release.
```

### 验收标准

- Quick Start 中所有命令真实可执行；
- 不出现“文档让用户拉取不存在镜像”的情况；
- 首页 CTA 与 Quick Start 保持一致。

+---------------------+
